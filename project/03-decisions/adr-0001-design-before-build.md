# ADR-0001: Design Before Build

## Context

The project started from a broad idea with many possible integrations and feature directions. Building immediately would have created high refactor risk and unclear MVP boundaries.

## Decision

Create and validate product and architecture documentation before substantial implementation work.

## Alternatives Considered

- Start coding immediately with a generic full-stack skeleton.
- Explore by prototyping most features before documenting scope.

## Consequences

- Slower time to first code change.
- Lower ambiguity and lower risk of uncontrolled feature creep.
- Better ability to create clean backlog items and implementation phases.
